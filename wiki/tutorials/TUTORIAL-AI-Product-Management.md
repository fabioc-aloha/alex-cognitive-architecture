## What You'll Build

A product management workflow using AI for PRDs, user research synthesis, roadmap planning, and stakeholder communication.

After this tutorial, you'll document decisions faster and communicate more clearly.

---

## 📋 Prerequisites

- Alex installed in VS Code
- A product area you're responsible for
- Access to user research or customer feedback

---

## Why AI-Assisted Product Management?

PMs spend too much time writing documents and not enough time thinking about products. AI helps with:

- Drafting PRDs from rough notes
- Synthesizing user research
- Creating roadmap presentations
- Stakeholder updates
- Competitive analysis

AI doesn't replace product sense. It amplifies it.

## 📍 Steps

### Step 1: Define Product Context

```
I'm PM for [product/feature area] at [company type].
Users: [describe primary users]
Business model: [how we make money]
Current priorities: [key initiatives]
My biggest challenge right now: [describe]
```

✅ **Checkpoint**: Alex understands your product context.

---

### Step 2: Write a PRD

Start with your rough idea:

```
I'm writing a PRD for [feature/product].
User problem: [what pain point are we solving]
Proposed solution: [rough description]
Success metrics: [how we'll measure success]
Constraints: [timeline, technical, resource]
```

Generate the full PRD:

```
Draft a PRD including:
- Problem statement
- User stories (3-5 key scenarios)
- Requirements (must have, should have, nice to have)
- Success metrics
- Risks and mitigations
- Open questions
```

Iterate section by section:

```
The user stories are too vague. 
Make them more specific with acceptance criteria.
```

---

### Step 3: Synthesize User Research

```
I have feedback from 12 user interviews about [topic]:
[paste notes or key quotes]

Synthesize this into:
- Top 3 themes
- Supporting evidence for each
- Contradictions or edge cases
- Recommendations
```

For survey data:

```
Here are open-ended responses from our NPS survey:
[paste responses]

Categorize the feedback and identify patterns.
```

---

### Step 4: Roadmap Communication

```
Create a roadmap presentation for [audience: leadership/team/customers].
Timeframe: [Q3 2026 / next 6 months]
Themes: [list major initiatives]
Tone: [strategic for execs, detailed for eng, benefit-focused for customers]
```

For trade-off discussions:

```
I need to present why we're deprioritizing [feature X].
The real reasons are [resource constraints, strategy shift, etc.].
Help me frame this constructively for stakeholders who wanted X.
```

---

### Step 5: Stakeholder Updates

```
Write a weekly product update for:
- What shipped last week
- What's in progress
- Blockers
- Upcoming decisions that need input

Keep it scannable — busy executives won't read paragraphs.
```

## 💡 Tips

### Competitive Analysis

```
Compare our [feature] to [competitor's feature].
Structure: feature comparison table, strengths/weaknesses, 
opportunities for differentiation.
```

### Sprint Planning

```
We have [X story points] capacity next sprint.
Here's the backlog: [paste items with estimates]
Help me prioritize based on: user impact, effort, dependencies.
```

### Metric Deep Dives

```
Our [metric] dropped 15% last week.
Possible causes: [list hypotheses]
Help me structure an investigation plan.
```

### Customer Meeting Prep

```
I have a call with [customer type] tomorrow.
They're considering [our product] for [use case].
What questions should I ask?
What objections should I prepare for?
```

## ⚠️ Common Mistakes

### PRDs Without User Evidence

AI can generate plausible-sounding PRDs for bad ideas. Always ground in:
- User research
- Data
- Business context

### Over-Documentation

Not everything needs a PRD:

```
Is this feature significant enough to need a full PRD?
Or would a 1-page brief suffice?
```

### Ignoring Politics

AI generates rational documents. Organizations are political:

```
I need to get [stakeholder] on board with this proposal.
Their concerns are usually about [X].
How can I address their perspective proactively?
```

## PM Document Templates

### Problem Brief

```
One-page problem definition:
1. Problem: [What user pain exists]
2. Evidence: [How we know this is real]
3. Impact: [Why it matters to the business]
4. Opportunity: [What solving it could unlock]
```

### Feature Spec

```
Detailed spec for engineering:
- User story
- Acceptance criteria
- Edge cases
- Dependencies
- Technical notes from previous discussions
```

### Launch Plan

```
Launch checklist:
- Marketing materials needed
- Documentation updates
- Support team training
- Metrics tracking setup
- Rollout strategy (% rollout, feature flags)
```
